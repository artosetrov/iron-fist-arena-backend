## Паттерны и уроки

### 1. Challenges route: 7 последовательных fix-коммитов (серийные ошибки сигнатур)

**Что произошло:** Коммиты `435ba0b` → `68114eb` → `553921b` → `6ef70b2` → `06582b4` → `6a64e1b` → `d925861` — каждый исправлял одну ошибку в challenges route: забытый `await`, неверное кол-во аргументов, несуществующие поля на `CombatResult`, отсутствие `prisma` как первого аргумента.

**Корневая причина:** Новый API route (`challenges/route.ts`) написан без предварительной проверки сигнатур game lib функций. Каждый `next build` находил одну ошибку → fix → rebuild → следующая ошибка.

**Правило уже есть** в CLAUDE.md ("Before using any function — check its signature") и в Oracle agent. Но оно не было соблюдено при написании нового route.

**Вывод:** Правило работает как документация, но не как превентивная мера. Сканер Oracle проверяет `await` но не проверяет количество аргументов или имена полей на результатах.

**Действие:** Нет изменений в правилах (они уже полные). Это вопрос дисциплины, а не отсутствия документации.

### 2. CodingKeys snake_case mismatch — третий инцидент

**Что произошло:** `72dc0fb` — модели в `Social.swift` и `Challenge.swift` имели explicit CodingKeys с snake_case маппингом, но бэкенд отправлял camelCase. Decode молча фейлился.

**Правило уже есть** (добавлено после второго инцидента). Список "affected models (fixed)" расширился.

**Вывод:** При создании НОВЫХ моделей для нового API (messaging) были правильно использованы identity CodingKeys (`Message.swift`). Значит правило усвоено для новых моделей, но старые модели (`Challenge.swift`) были написаны до правила.
